# Zip Game Simulation Plan

This plan is for building a playable LinkedIn Zip-style puzzle simulator. The current goal is the simulation itself: puzzle data, movement rules, puzzle generation, solver verification, rendering, and a mouse-playable UI.

Future agent-training work belongs in `training_plan.md`; do not build it as part of the simulator milestone.

---

## 1. Core Data Model

### Immutable Puzzle Definition

`Puzzle` should contain only data that does not change during play:

- `rows: int`
- `cols: int`
- `waypoints: list[tuple[int, int]]`
  - Ordered positions for waypoint labels `1, 2, 3, ...`.
  - `waypoints[0]` is the start cell.
  - `waypoints[-1]` is the final required waypoint.
- `walls: set[tuple[tuple[int, int], tuple[int, int]]]`
  - Canonical representation of blocked edges.
  - Store each wall as a sorted pair of adjacent coordinates.
  - Derive per-cell wall flags only when rendering or checking moves.
- `solution: list[tuple[int, int]] | None`
  - Ground-truth Hamiltonian path when generated by this project.
  - Optional for externally loaded puzzles.
- `difficulty: str`
- `seed: int | None`
- `unique_solution: bool | None`
  - `True`, `False`, or `None` if uniqueness was not checked or timed out.

### Mutable Game State

`GameState` should contain only runtime state:

- `path: list[tuple[int, int]]`
- `visited: set[tuple[int, int]]` or a bitset for performance
- `current_pos: tuple[int, int]`
- `next_waypoint_index: int`
- `done: bool`
- `success: bool`
- `step_count: int`

This separation is important for:

- Reset correctness
- UI undo/reset
- Replay and playback
- Puzzle save/load without runtime state leaks
- Future automation without rewriting the simulator

### Serialization

Use JSON for puzzle files:

```json
{
  "rows": 6,
  "cols": 6,
  "waypoints": [[0, 0], [2, 4], [5, 5]],
  "walls": [[[0, 1], [0, 2]]],
  "solution": [[0, 0], [0, 1]],
  "difficulty": "medium",
  "seed": 123,
  "unique_solution": true
}
```

Use a small serialization helper instead of ad hoc JSON code spread through the project.

## 2. Movement Rules

- Movement is 4-directional only: up, down, left, right.
- The path cannot revisit any cell.
- The path must visit every cell exactly once.
- The player starts at waypoint `1`.
- Waypoints must be visited in strict ascending order.
- The player cannot step onto a future waypoint early.
- The player cannot skip a waypoint.
- Walls block movement between two adjacent cells in both directions.
- Non-waypoint cells can be used freely between consecutive waypoints.
- A puzzle is solved when every cell is visited and the final waypoint has been reached in order.

Implementation detail: the rule checker should be one shared function used by the solver, simulator, and UI. Do not duplicate move legality logic in multiple files.

## 3. Simulator API

The simulator should expose a small API that the UI can call directly:

- `new_game(puzzle: Puzzle) -> GameState`
- `legal_moves(puzzle: Puzzle, state: GameState) -> list[Direction]`
- `can_move(puzzle: Puzzle, state: GameState, direction: Direction) -> bool`
- `step(puzzle: Puzzle, state: GameState, direction: Direction) -> StepResult`
- `undo(state: GameState) -> GameState`
- `reset(puzzle: Puzzle) -> GameState`
- `is_solved(puzzle: Puzzle, state: GameState) -> bool`

`StepResult` should include:

- Updated state
- Whether the move was valid
- Failure reason if invalid
- Whether the puzzle is solved

Invalid moves should be reported cleanly so the UI can show feedback without crashing or corrupting state.

## 4. Puzzle Generation

Generate the solution first, then derive the visible puzzle from it. Do not place waypoints on an empty grid and hope the puzzle is solvable.

### Step 1: Generate a Hamiltonian Path

Use randomized DFS with backtracking:

1. Pick a start cell.
2. Prefer starts on edges or corners for harder difficulties.
3. Shuffle valid unvisited neighbors.
4. Move to a neighbor, mark it visited, and continue.
5. Backtrack when stuck.
6. Stop when all `rows * cols` cells are in the path.

For larger grids, use:

- Iterative DFS instead of recursion
- Deterministic random seeds
- Attempt limits
- Timeouts
- Flood-fill pruning to avoid isolating unvisited regions
- Neighbor ordering that prefers moves with fewer onward choices

If generation exceeds the attempt or time budget, restart with a new seed.

### Step 2: Place Waypoints

Given a solution path of `num_cells = rows * cols`, choose `num_waypoints` indices along that path.

Always include:

- Index `0`
- Index `num_cells - 1`

Difficulty controls waypoint density:

- Super easy: waypoints every 2-3 cells
- Easy: waypoints every 4-6 cells
- Medium: waypoints every 6-10 cells
- Hard: 4-6 total waypoints
- Expert: 2-3 total waypoints

Waypoint placement can be even, randomized within bands, or biased toward interesting turns and bottlenecks.

### Step 3: Add Walls

Walls may only be placed on edges the solution path does not use.

Good candidate walls:

- Adjacent cells that are not consecutive in the solution path
- Tempting wrong moves that jump between far-apart parts of the path
- Edges that reduce ambiguity without making the puzzle trivial

Important: more walls do not always mean harder. Some walls make puzzles easier by eliminating wrong choices.

### Step 4: Verify Solvability and Uniqueness

Run the standalone solver after generation.

Validation should record:

- `solution_count`
- `unique_solution`
- `solver_timed_out`
- `generation_seed`
- `generation_attempts`

For medium and above, require uniqueness unless intentionally generating multi-solution puzzles for experimentation.

## 5. Difficulty Config

`DifficultyConfig` should include all generation knobs:

| Level | Grid Size | Waypoints | Walls | Unique Solution |
| --- | --- | --- | --- | --- |
| Super Easy | 4x4 | Every 2 cells | 0 | Not required |
| Easy | 5x5 | Every 4 cells | 2-4 | Preferred |
| Medium | 6x6 | Every 6-8 cells | 4-8 | Required |
| Hard | 7x7 or 8x8 | 4-6 total | 6-12 | Required |
| Expert | 8x8 to 10x10 | 2-3 total | 8-20 | Required |

Additional config fields:

- `max_generation_attempts`
- `generation_timeout_seconds`
- `solver_timeout_seconds`
- `allow_multi_solution`
- `start_position_policy`
- `waypoint_spacing_policy`
- `wall_candidate_policy`
- `seed`

## 6. Standalone Solver

The solver is separate from both the UI and the simulator state transitions. It is used for:

- Verifying solvability
- Verifying uniqueness
- Producing a known solution path for generated puzzles
- Evaluating generated puzzle quality
- Debugging the UI and simulator

### Algorithm

Use iterative DFS with pruning:

1. Start from waypoint `1`.
2. Maintain `(position, visited_mask, next_waypoint_index, path)`.
3. Expand legal neighbors only.
4. Forbid revisits, walls, out-of-bounds moves, and early waypoint visits.
5. Stop after finding two complete solutions when only uniqueness is needed.

Performance improvements:

- Represent visited cells as a bitmask.
- Precompute neighbor lists.
- Precompute wall-blocked edges.
- Use flood-fill pruning for unvisited cells.
- Enforce waypoint reachability constraints.
- Add a timeout and return `unknown` if exceeded.

Note: every complete valid solution has path length `rows * cols - 1`, so the solver is mainly for existence, uniqueness, and solution recovery rather than shortest-path optimization.

## 7. Rendering and Mouse-Playable UI

The UI is a core part of the simulator. It should be good enough to actually play the game with a mouse.

### Render Modes

Implement:

- `ansi`: text rendering for logs and tests
- `image`: static image rendering for screenshots/debugging
- `human`: interactive window

Pygame is a good first choice because it is simple, fast, and works well for grid interactions. A web UI can be added later if needed.

### Board Rendering

The board should clearly show:

- Grid cells
- Numbered waypoints
- Walls between cells
- Current path
- Current head
- Legal next cells
- Invalid move feedback
- Solved state

Use stable dimensions so text, walls, highlights, and path segments do not shift the layout.

### Mouse Interaction

The UI should support:

- Click a legal adjacent cell to extend the path
- Drag through legal adjacent cells to draw the path
- Undo last move
- Reset puzzle
- Generate a new puzzle by difficulty
- Load a saved puzzle JSON
- Save the current puzzle JSON
- Highlight legal next moves on hover
- Show a clear visual response for invalid moves

Coordinate mapping from mouse position to `(row, col)` should be covered by tests because it is easy to get subtly wrong.

### Playback and Debugging

The simulator UI should also support playback:

- Replay the known solution path
- Step forward and backward through a path
- Pause/resume playback
- Speed control
- Compare the player's current path to the stored solution when available
- Show puzzle metadata such as difficulty, seed, uniqueness, and visited count

These features are useful now for debugging generated puzzles and later for observing trained agents.

## 8. Testing Plan

Add tests before the system gets large.

### Unit Tests

- Coordinate conversion and bounds checks
- Neighbor generation
- Wall canonicalization
- Bidirectional wall blocking
- Movement legality
- Waypoint ordering
- Revisit prevention
- Puzzle serialization round trip
- Game reset behavior
- Undo behavior

### Solver Tests

- Finds a known solution
- Rejects impossible puzzles
- Detects multiple solutions
- Stops at two solutions for uniqueness checks
- Handles timeout cleanly

### Generator Tests

- Generated solution visits every cell exactly once
- Waypoints lie on the solution path
- Walls never block solution edges
- Generated puzzles are solvable
- Required difficulties enforce uniqueness

### UI Tests

- Pixel-to-cell coordinate mapping
- Click and drag path creation
- Undo/reset behavior
- Playback state transitions

## 9. Suggested Module Structure

```text
zip_game/
├── __init__.py
├── config.py          # DifficultyConfig and UI config
├── puzzle.py          # Immutable Puzzle model and serialization
├── state.py           # Mutable GameState model
├── rules.py           # Shared legality and transition checks
├── simulator.py       # Public simulation API: step, undo, reset, legal moves
├── generator.py       # Hamiltonian path, waypoint, and wall generation
├── solver.py          # Solvability, uniqueness, and solution recovery
├── renderer.py        # ansi and image renderers
├── ui.py              # Mouse-playable UI
└── utils.py           # Coordinate, adjacency, and bitmask helpers
```

Keep `rules.py` central. The solver, simulator, and UI should not each invent their own legality rules.
