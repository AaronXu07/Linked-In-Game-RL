# Patches Game Simulation Plan

This plan is for building a playable LinkedIn Patches-style puzzle simulator. The
current goal is the simulation itself: puzzle data, placement rules, puzzle
generation, solver verification, rendering, and a mouse-playable UI.

Future agent-training work belongs in a separate `training_plan.md`; do not build
it as part of the simulator milestone.

---

## 0. Game Overview

Patches is a spatial logic puzzle. You partition a rectangular grid into
non-overlapping rectangular regions ("patches") so that:

- Every patch contains **exactly one** clue cell.
- Every clue's **number** equals the **area** (cell count) of its patch.
- Every clue's **shape type** is satisfied by its patch's dimensions.
- Every grid cell is covered by **exactly one** patch (no gaps, no overlaps).

The puzzle is solved when the whole grid is partitioned this way. Well-formed
puzzles have exactly one logically deducible solution.

### Clue shape types

Each clue carries a shape constraint on its patch's width `w` and height `h`
(where `w * h == number`):

- `SQUARE`: `w == h` (so `number` must be a perfect square).
- `WIDE`: `w > h`.
- `TALL`: `h > w`.
- `FREE` (a.k.a. any): any `w x h` whose area equals `number`; no aspect
  constraint. A single cell (`number == 1`) is trivially valid for `FREE` and,
  by convention, for `SQUARE` (1x1).

Keep shape validation in one shared helper so the solver, simulator, and UI all
agree on edge cases (especially `number == 1` and prime `number` under
`SQUARE`).

## 1. Core Data Model

### Immutable Puzzle Definition

`Puzzle` should contain only data that does not change during play:

- `rows: int`
- `cols: int`
- `clues: list[Clue]`
  - Each `Clue` has:
    - `pos: tuple[int, int]` — the clue cell `(row, col)`.
    - `number: int` — required patch area (>= 1).
    - `shape: ShapeType` — `SQUARE | WIDE | TALL | FREE`.
    - `id: int` — stable index used to color/identify the patch.
  - Invariant: `sum(clue.number for clue in clues) == rows * cols`.
  - Invariant: clue positions are distinct and in bounds.
- `solution: list[Rect] | None`
  - Ground-truth partition when generated by this project.
  - `Rect` is `(top, left, height, width)` and maps to exactly one clue id.
  - Optional for externally loaded puzzles.
- `difficulty: str`
- `seed: int | None`
- `unique_solution: bool | None`
  - `True`, `False`, or `None` if uniqueness was not checked or timed out.

### Mutable Game State

`GameState` should contain only runtime state:

- `assignment: dict[tuple[int, int], int]`
  - Maps each covered cell to the clue id of the patch covering it.
  - Absence of a cell means "unassigned".
- `patches: dict[int, Rect]`
  - Current committed rectangle for each clue id (may be partial/absent).
- `history: list[Move]` — for undo/redo.
- `done: bool`
- `success: bool`
- `step_count: int`

This separation matters for reset correctness, UI undo/reset, replay, and
save/load without runtime state leaking into the puzzle definition.

### Serialization

Use JSON for puzzle files:

```json
{
  "rows": 6,
  "cols": 6,
  "clues": [
    {"id": 0, "pos": [0, 0], "number": 4, "shape": "square"},
    {"id": 1, "pos": [0, 3], "number": 6, "shape": "wide"},
    {"id": 2, "pos": [3, 0], "number": 6, "shape": "tall"},
    {"id": 3, "pos": [5, 5], "number": 20, "shape": "free"}
  ],
  "solution": [
    {"clue_id": 0, "top": 0, "left": 0, "height": 2, "width": 2}
  ],
  "difficulty": "medium",
  "seed": 123,
  "unique_solution": true
}
```

Use a small serialization helper instead of ad hoc JSON code spread through the
project. Round-trip (`to_dict`/`from_dict`) must be lossless.

## 2. Placement Rules

A single shared rule module defines validity. Do not duplicate this logic in the
solver, simulator, and UI.

- A patch is an axis-aligned rectangle of grid cells.
- A patch must contain exactly one clue cell.
- A patch's area must equal its clue's `number`.
- A patch's dimensions must satisfy the clue's shape type.
- Patches may not overlap: a cell belongs to at most one patch.
- The board is solved when every cell belongs to exactly one valid patch and
  every clue is used exactly once.

Helper predicates to expose:

- `rect_cells(rect) -> Iterable[cell]`
- `rect_in_bounds(rows, cols, rect) -> bool`
- `rect_contains_single_clue(puzzle, rect) -> bool`
- `shape_ok(shape, height, width) -> bool`
- `area_ok(number, height, width) -> bool`
- `rect_conflicts(state, rect) -> bool` (overlaps an existing patch)
- `is_solved(puzzle, state) -> bool`

Invalid placements should be reported cleanly with a reason so the UI can show
feedback without crashing or corrupting state.

## 3. Simulator API

The simulator exposes a small API the UI can call directly:

- `new_game(puzzle: Puzzle) -> GameState`
- `candidate_rects(puzzle, clue_id) -> list[Rect]`
  - All rectangles that are area/shape/bounds/clue-valid for a clue, ignoring
    current occupancy. Useful for UI hints and the solver.
- `can_place(puzzle, state, clue_id, rect) -> PlacementResult`
- `place(puzzle, state, clue_id, rect) -> StepResult`
  - Commits (or replaces) the patch for `clue_id`.
- `clear_patch(state, clue_id) -> GameState`
- `undo(state) -> GameState`
- `reset(puzzle) -> GameState`
- `is_solved(puzzle, state) -> bool`

`StepResult` / `PlacementResult` should include:

- Updated state
- Whether the placement was valid
- Failure reason if invalid (out of bounds, wrong area, wrong shape, overlap,
  wrong/zero/multiple clues enclosed)
- Whether the puzzle is now solved

## 4. Puzzle Generation

Generate a valid full partition first, then derive the visible clues from it. Do
not scatter clues on an empty grid and hope for a unique tiling.

### Step 1: Generate a Rectangle Tiling

Partition `rows x cols` into rectangles that fully cover the grid with no
overlaps:

1. Maintain a set of uncovered cells.
2. Pick the top-left-most uncovered cell.
3. Choose a random rectangle rooted at that cell whose size fits the remaining
   space and respects a per-difficulty max area.
4. Prefer rectangle dimensions that leave the remaining region tileable
   (avoid creating thin unfillable slivers).
5. Commit the rectangle and repeat until the grid is covered.
6. Backtrack when a placement makes the remainder untileable.

For larger grids use:

- Iterative placement with backtracking (not deep recursion)
- Deterministic random seeds
- Attempt limits and timeouts
- A remainder-tileability check (flood fill / min-dimension pruning)

If generation exceeds the attempt or time budget, restart with a new seed.

### Step 2: Derive Clues From The Tiling

For each rectangle in the tiling:

- Choose one cell inside it as the clue cell (center-ish or random per policy).
- `number = height * width`.
- Assign a `shape` consistent with the rectangle:
  - `w == h` -> may be `SQUARE` (or `FREE`).
  - `w > h` -> may be `WIDE` (or `FREE`).
  - `h > w` -> may be `TALL` (or `FREE`).
- Difficulty controls how often `FREE` is used instead of the tight shape.
  Tight shapes constrain the search and make puzzles easier; more `FREE` clues
  make puzzles harder and can threaten uniqueness.

### Step 3: Verify Uniqueness

Run the standalone solver against the derived clues (without the stored
solution). Record:

- `solution_count` (stop counting at 2)
- `unique_solution`
- `solver_timed_out`
- `generation_seed`
- `generation_attempts`

If a puzzle is not unique and uniqueness is required, either tighten a `FREE`
clue to its concrete shape, nudge a clue position, or regenerate.

## 5. Difficulty Config

`DifficultyConfig` should include all generation knobs:

| Level      | Grid Size    | Patches (approx) | Max Patch Area | FREE-shape ratio | Unique Solution |
| ---------- | ------------ | ---------------- | -------------- | ---------------- | --------------- |
| Super Easy | 5x5          | many, small      | 4              | low              | Not required    |
| Easy       | 6x6          | small-medium     | 6              | low              | Preferred       |
| Medium     | 7x7          | medium           | 9              | medium           | Required        |
| Hard       | 8x8          | mixed            | 12             | medium-high      | Required        |
| Expert     | 9x9 to 10x10 | few, large       | 16+            | high             | Required        |

Additional config fields:

- `max_generation_attempts`
- `generation_timeout_seconds`
- `solver_timeout_seconds`
- `allow_multi_solution`
- `clue_position_policy` (center vs random within rectangle)
- `shape_tightness_policy` (how often to emit `FREE`)
- `min_patch_area` / `max_patch_area`
- `seed`

## 6. Standalone Solver

The solver is separate from the UI and simulator transitions. It is used for
verifying solvability, verifying uniqueness, recovering a solution path for
generated puzzles, evaluating puzzle quality, and debugging.

### Algorithm

Model as exact cover / constrained backtracking:

1. Precompute, for each clue, its list of candidate rectangles (bounds + area +
   shape + encloses only that clue).
2. Pick the most-constrained uncovered cell (fewest candidate rectangles that
   could cover it) — a "minimum remaining values" heuristic.
3. Try each candidate rectangle covering that cell:
   - Skip if it overlaps already-placed patches.
   - Place it, mark covered cells, recurse.
   - Undo on failure.
4. Succeed when all cells are covered and all clues used exactly once.
5. Stop after finding two solutions when only uniqueness is needed.

Performance improvements:

- Represent covered cells as a bitmask over `rows * cols`.
- Precompute candidate rectangles per clue once.
- Prune candidates whose cells would strand an uncoverable cell.
- Add a timeout and return `unknown` if exceeded.

## 7. Rendering and Mouse-Playable UI

The UI is a core part of the simulator. It should be good enough to actually
play the game with a mouse.

### Render Modes

- `ansi`: text rendering for logs and tests (clue numbers + shape glyphs, patch
  borders drawn with box characters).
- `image`: static image rendering for screenshots/debugging (dependency-free
  PPM is fine, matching the Zip project).
- `human`: interactive Pygame window.

### Board Rendering

The board should clearly show:

- Grid cells and coordinates
- Clue cells with their number and a shape indicator (square/wide/tall/free)
- Committed patches, each filled with a distinct stable color
- The patch currently being dragged (preview rectangle)
- Valid vs invalid preview feedback (e.g., green vs red outline)
- Uncovered cells
- Solved state

Use stable dimensions so numbers, borders, highlights, and fills do not shift
the layout.

### Mouse Interaction

Core interaction mirrors the real game:

- Press on (or near) a clue, drag outward to form a rectangle, release to commit.
- Live preview shows current area vs required number and whether the shape is
  satisfied.
- Click a committed patch to clear it.
- Undo last action; Reset puzzle.
- Generate a new puzzle by difficulty.
- Load / Save puzzle JSON.
- Optional: highlight the unique valid rectangle when a clue has only one.

Coordinate mapping from mouse position to `(row, col)` and from a drag to a
`Rect` should be covered by tests because it is easy to get subtly wrong.

### Playback and Debugging

- Reveal / replay the stored generated solution patch-by-patch.
- Step forward and backward through the solution.
- Show puzzle metadata: difficulty, seed, uniqueness, covered-cell count.
- Compare the player's current partition to the stored solution when available.

## 8. Testing Plan

Add tests before the system gets large.

### Unit Tests

- Coordinate conversion and bounds checks
- `Rect` cell enumeration and drag-to-rect conversion
- Area/shape validity (including `number == 1`, primes under `SQUARE`)
- Single-clue-enclosure checks
- Overlap detection
- Solved-state detection
- Puzzle serialization round trip
- Reset and undo behavior

### Solver Tests

- Finds a known solution for a generated puzzle
- Rejects impossible clue sets (area sum mismatch, unsatisfiable shape)
- Detects multiple solutions
- Stops at two solutions for uniqueness checks
- Handles timeout cleanly

### Generator Tests

- Tiling fully covers the grid with no overlaps
- Clue areas sum to `rows * cols`
- Each clue lies inside its own rectangle
- Assigned shapes are consistent with their rectangles
- Generated puzzles are solvable
- Required difficulties enforce uniqueness

### UI Tests

- Pixel-to-cell mapping
- Drag start/end to `Rect` conversion
- Commit / clear / undo / reset behavior
- Playback state transitions

## 9. Suggested Module Structure

```text
patches/simulation/
├── __init__.py
├── config.py          # DifficultyConfig and UI config
├── puzzle.py          # Immutable Puzzle + Clue models and serialization
├── state.py           # Mutable GameState, Move, StepResult
├── rules.py           # Shared area/shape/overlap/solved checks
├── simulator.py       # Public API: new_game, place, clear, undo, reset
├── generator.py       # Tiling generation, clue derivation, shape assignment
├── solver.py          # Exact-cover backtracking solver + uniqueness
├── renderer.py        # ansi and image renderers
├── ui.py              # Mouse-playable Pygame UI
└── utils.py           # Coordinates, Rect helpers, ShapeType, bitmask helpers
```

Keep `rules.py` central. The solver, simulator, and UI must not each invent
their own validity rules.
